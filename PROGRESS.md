## Phase 2 verification notes (2026-07-21)
- All 4 chaos scenarios were run live end-to-end (`chaos/scenarios.py`) against the real stack multiple times while iterating; final clean pass used `SCENARIO_DURATION=180` (see `tests/test_detection_scenarios.py`) because real checkout traffic is too sparse for 60-120s windows to reliably contain an order (see known issue above).
- **Bug found and fixed ŌĆö Prometheus rate-window too narrow.** Assumed the collector exports `traces_span_metrics_*`/`kafka_message_count_total` roughly every ~15s (matching typical scrape cadence); empirically it's actually ~60s. A `rate(...[1m])` lookback frequently contained <2 real samples depending on query alignment and Prometheus silently returned *zero result rows* (not a zero value) ŌĆö intermittent on span metrics, reliable on the Kafka counter. Fixed by widening to `rate(...[2m])` in `retrieval/metric_client.py` (`_RATE_WINDOW`, `_KAFKA_RATE_WINDOW`). A window that's too wide has its own cost (dilutes a short real burst back down near baseline ŌĆö tried `5m` for Kafka first, it masked the flood); `2m` is the minimum with a comfortable margin over the observed ~60s cadence.
- **Bug found and fixed ŌĆö baseline mean corrupted by real outliers.** `ad`'s periodic GC-pause blips (above) sit inside `get_baseline`'s 30-min lookback and dragged a *mean* baseline high enough (~1700ms) to mask a genuine, smaller overload-scenario spike (~600-900ms) that was still 20-200x its *true* typical latency (~4-40ms). Switched `MetricClient.get_baseline` to `statistics.median` ŌĆö median ignores the rare huge samples and reflects the service's typical value.
- **Design change ŌĆö payment failures need an HTTP-log-based error check, not span status.** Confirmed live: `paymentFailure`/`paymentUnreachable` do NOT set span `status=ERROR` even on a real HTTP 500 from `/api/checkout` (matches the Phase-1 finding below), so the generic per-service `traces_span_metrics_calls_total{status_code=ERROR}` check never fires for these two scenarios. Added `SignalDetector._check_checkout_http_error_rate`, which parses `frontend-proxy`'s envoy access log lines from Loki (`"POST /api/checkout HTTP/1.1" 500 ...`) directly ŌĆö this is the only place the failure is visible for these two scenarios. (Note: `payment_outage`'s `paymentUnreachable` *does* also produce real `STATUS_CODE_ERROR` span-metric entries, since a refused gRPC connection is a transport-level error the client library flags itself, unlike the app-level "declined" case ŌĆö so `payment_outage` fires `error_rate_spike` two ways, `payment_failure` only via the HTTP-log path.)
- **`detection/service_map.yaml` trimmed to just `checkout: [...]`.** Dropped the `frontend: [cart, checkout, product-catalog, recommendation, ad]` entry from the original spec sketch ŌĆö an ordinary frontend-rooted trace (e.g. browsing a single product) legitimately doesn't call all of cart/checkout/recommendation/ad together, so requiring all of them per-trace is a guaranteed false positive on normal traffic, not a useful check.
- **Kafka publish-vs-consume ratio dropped as a firing condition** (kept in `metric_values` for context only). `kafka_message_count_total` (broker-wide) and `kafka_consumer_records_consumed_rate` (fraud-detection's self-reported rate) are on different scales and sit ~9-30x apart even in a verified-quiet window with all flags off ŌĆö not usable as a ratio threshold in this environment. `queue_anomaly` now fires only on publish-rate-vs-its-own-30-min-baseline (median), which is scale-consistent and was confirmed to fire correctly (and not fire when quiet).
- Final live-scenario results (all pass, `pytest tests/test_detection_scenarios.py -v -s`, `SCENARIO_DURATION=180`): `payment_failure`ŌåÆ`error_rate_spike`+`span_gap`+`throughput_drop`; `payment_outage`ŌåÆ`error_rate_spike`(3 services)+`span_gap`; `queue_backlog`ŌåÆ`queue_anomaly` (verified via direct replay against the historical window with the final code, after the rate-window fix); `overload`ŌåÆ`latency_spike` on `ad` (verified via direct replay against the historical window with the final code, after the median-baseline fix). See git history for the exact printed signal tables during iteration.
