- **Container resource metric names (Phase 3 finding)**: NOT cAdvisor-style (`container_cpu_usage_seconds_total` etc. don't exist here). This demo's Prometheus exposes `container_cpu_utilization_ratio` (label `container_name`; % of one host core, unbounded — confirmed hitting ~374% on `ad` during a real `adHighCpu` run) and `container_memory_percent_ratio` / `container_memory_usage_total_bytes` / `container_memory_usage_limit_bytes` (also keyed by `container_name`, matches each service's real `deploy.resources.limits.memory`, confirmed byte-exact for `ad`: 314572800 = 300MiB). **No container in this compose has a real CPU quota** (`docker inspect` shows `NanoCpus=0`/`CpuQuota=0` on every service, base compose only sets memory limits) — `cpu_high`'s 80% threshold is therefore against a nominal one-core budget, not a real configured limit, unlike `memory_high`'s 90% which is real. `retrieval/metric_client.py`'s `RESOURCE_SERVICES` list is the 16 app/business containers (excludes prometheus/grafana/tempo/loki/otel-collector/jaeger/flagd/flagd-ui/kafka/postgresql/valkey-cart/load-generator/llm).
- **Chronic non-chaos resource condition confirmed still present in Phase 3**: `ad` and `fraud-detection` both sit ~99-100% of their 300MiB memory limit continuously (real, not a chaos artifact — same root cause as the Phase 2 known issue on `ad`'s GC pauses). This means the Phase 3 `scheduled` trigger path's "no signals → zero LLM calls" branch will rarely actually fire in this environment — `memory_high` on `ad`/`fraud-detection` is basically always present. Treated as correct, honest behavior (there IS a real resource condition worth surfacing), not a detector bug to suppress.
- **Real traffic levels** (this env, load-generator's Playwright bug reduces effective load): checkout ~0.25-0.4 req/s (~1.4 orders/min), frontend ~2 req/s, product-catalog ~0.9-1.1 req/s, ad/recommendation/payment/shipping/currency all <0.1 req/s. Windows under ~180s can easily see zero events for the low-traffic services.
