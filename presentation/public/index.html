08 · Retrieval strategy

Structured live retrieval, deliberately not vector RAG

"RAG" here means retrieval-augmented generation over live typed telemetry — not embeddings over a document corpus. That was a considered rejection, not an omission.

Vector RAG — considered, rejected

  • Embed logs/metrics, retrieve by similarity
  • Adds indexing latency and staleness for data that changes every few seconds
  • Relevant evidence is structural (service name, time range) — not semantic
  • Needs an extra vector-DB service to run and keep fresh

What this system does instead

  • Typed queries straight against Loki / Prometheus / Tempo, by service + time range
  • Always current — fetched fresh at answer time, never a stale index
  • Deterministic mode-routing decides what to fetch — no relevance scoring needed
  • One fewer service to run, deploy, and keep in sync
Design decision, recorded verbatim: "explicitly NOT a vector DB" — see project decision log.

10 · Deterministic detection layer

Eight signal checks, pure Python, zero LLM cost

The AI explains anomalies this layer already measured — it never has to spot one from raw numbers alone.

latency_spike

p95 vs. rolling baseline, requires ≥2 samples — one noisy bucket can't fire alone.

error_rate_spike

Absolute floor OR relative-to-baseline jump, whichever fires.

checkout_http_error_rate

HTTP 5xx fallback — catches failures the span status itself can't show.

span_gap

Diffs a real trace tree against the expected service map — flags missing children.

queue_anomaly

Kafka publish-rate vs. baseline — not consumer lag, which stayed at 0 in real testing.

throughput_drop

Request rate falls below a baseline ratio.

cpu_high

Utilization vs. container limit, not raw usage.

memory_high

Same, for memory — feeds both alerts and right-sizing answers.

One shared mechanic — baseline comparison — repeated across all 8 checks.

signal_typestringe.g. latency_spike, cpu_high, span_gap

Called from three places, same trusted output

SignalDetector.detect(start, end) runs all 8 checks and returns every signal that fired. The scheduled scanner polls it every 60s, the dashboard's /alerts calls it per request, and the orchestrator injects its output into build_context() as measured, trusted facts — the LLM never has to re-derive them.

12 · From evidence to root cause

Findings are the structured unit of hypothesis

The Finding schema every hypothesis must fit before it can become an answer.

categoryenumerror · performance · resource · capacity · healthy
servicestring?which service this observation is about, if any
summarystringone-sentence observation
evidencelist[string]quoted values that must exist in the fetched context
severityenumcritical · warning · info

Root cause = synthesis, not detection

The detector already found that something's wrong (trusted signal). The LLM's job is narrower and harder: turn several Findings plus raw logs/metrics into one causal sentence — and the answer field must stand alone with real numbers in it, not just point at findings.

13 · Recommendation generation

Recommendations are scored by blast radius, not urgency

The Recommendation schema — and the one hard rule wired into the system prompt itself.

actionstringa reviewable config / scaling / code change
rationalestringwhy, tied back to the cited evidence
risk_levelenumlow · medium · high — monitoring vs. changing core logic
referenceslist[string]populated only when web search cited an external source

🚫 Never destructive, by construction

requires_human_review is always true. The system prompt explicitly bars destructive actions — no data deletion, no uncontrolled restarts — the model is not trusted to self-certify safety, the schema and prompt enforce it.

17 · Grounding safeguards

Every fix below lives in retrieval code, never in a prompt tweak

Hallucination bugs were found by cross-checking answers against ground truth, then fixed structurally.

🧩 Cross-service log mapping

Named-service questions also pull the services where the real reason actually lives.

📏 Per-service log capping

Each service's log contribution is capped independently — no service can crowd out another's evidence.

🔁 Fresh data per comparison

Multi-service questions fetch live metrics for every named service, not just the first match.

📐 Labeled rows, not raw JSON

Dense nested dicts caused real cross-service misattribution in testing; explicit labeled rows fixed it.

18 · Takeaways

Three things to remember

01

Deterministic where possible

Detection is pure Python and free to run. The LLM is reserved for the one step that actually needs reasoning: synthesis.

02

Retrieval is the product

Grounding lives in build_context(), not the prompt. Mode-routing, cross-service mapping, and per-service capping do the real anti-hallucination work.

03

Validated against ground truth

Every claim in this deck, and every claim the AI makes at runtime, is checked against real Loki/Prometheus/Tempo data — not asserted.
